# プロジェクト準拠ガイドライン

## 最上位原則

**すべての作業はプロジェクトの作法に合わせて行うこと。これは絶対のルールである。**

グローバルガイドライン（`~/.claude/rules/` 配下）は、プロジェクト固有のルールが存在しない場合のフォールバックとしてのみ適用する。

## 優先順位

```
1. プロジェクト固有の設定・規約（最優先・絶対）
2. モジュール/ディレクトリ固有の設定
3. グローバルガイドライン（フォールバック）
```

## 準拠すべき項目

### コーディングルール

- プロジェクトの AGENTS.md / CLAUDE.md / CONTRIBUTING.md に記載のルールに従う
- プロジェクトの `.editorconfig`、リンター設定（ESLint, Flake8 等）に従う
- プロジェクトの Prettier / Black 等のフォーマッター設定に従う
- **プロジェクト内の既存コードのスタイルを踏襲する**（命名規則、インデント、括弧スタイル等）

### コメントの書き方

- プロジェクト内の既存コメントのスタイル（言語、形式、粒度）に合わせる
- JSDoc / docstring / 型注釈の有無・スタイルはプロジェクトの慣例に従う
- プロジェクトがコメントを書かない方針なら、不要なコメントを追加しない

### ビルド・型チェック・テスト

- プロジェクトで定義されたコマンドを使用する
- コマンドの確認先（優先順）:
  1. AGENTS.md / CLAUDE.md / README.md に記載のコマンド
  2. `package.json` の `scripts`
  3. `Makefile` / `Taskfile.yml` / `justfile`
  4. `pyproject.toml` / `setup.cfg` の設定
  5. CI設定ファイル（`.github/workflows/` 等）
- **グローバルで定義したコマンドを勝手に適用しない**（例: プロジェクトが `yarn test` を使っているのに `npm test` を実行しない）

### 依存管理

- プロジェクトのパッケージマネージャーに従う（npm / yarn / pnpm / bun 等）
- ロックファイルの種類で判断: `package-lock.json` → npm、`yarn.lock` → yarn、`pnpm-lock.yaml` → pnpm、`bun.lockb` → bun

### ディレクトリ構造

- 新規ファイルはプロジェクトの既存構造に合わせて配置する
- プロジェクトの命名規則に従う（kebab-case, camelCase, PascalCase 等）

## 作業開始時の確認フロー

プロジェクトで作業を開始する際、以下を確認する:

1. **プロジェクトのルールファイルを読む**
   - `AGENTS.md` / `CLAUDE.md` / `CONTRIBUTING.md` / `README.md`
2. **設定ファイルを確認する**
   - リンター・フォーマッター設定、`package.json` の `scripts` 等
3. **既存コードのパターンを把握する**
   - 命名規則、ディレクトリ構造、コメントスタイル
4. **プロジェクトにルールがない項目のみ**、グローバルガイドラインを適用する

## 禁止事項

- プロジェクトの規約と矛盾するグローバルルールを適用すること
- プロジェクトのコマンド体系を無視して独自のコマンドを実行すること
- プロジェクトのコードスタイルと異なるスタイルでコードを書くこと
- プロジェクトが採用していないツール・ライブラリを勝手に導入すること
